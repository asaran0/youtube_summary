Q1: A developer accidentally pushed buggy code to the main branch. How would you handle it?

A: If the code is already pushed and other developers may have pulled it, I would use `git revert` to create a new commit that undoes the changes.

```bash
git revert <commit-id>
git push origin main
```

I avoid `git reset` in shared branches because it rewrites history and can affect other developers.

---

Q2: You made changes in the wrong branch. How would you move those changes to the correct branch?

A: First, I would save my changes using `git stash`, switch to the correct branch, and then apply the changes.

```bash
git stash
git checkout correct-branch
git stash pop
```

Alternatively, if the changes are already committed, I can use `git cherry-pick`.

---

Q3: You committed a file but forgot to include another file in the same commit. What would you do?

A: I would add the missing file and amend the previous commit.

```bash
git add missing-file.txt
git commit --amend
```

This updates the last commit instead of creating a new one.

---

Q4: You need to work on a new feature, but your current work is incomplete. What would you do?

A: I would temporarily save my unfinished work using `git stash`, switch to the new branch, and continue later.

```bash
git stash
git checkout feature-branch
```

When I return:

```bash
git stash pop
```

---

Q5: Two developers modified the same file, and a merge conflict occurred. How would you resolve it?

A: I would:

1. Open the conflicted file.
2. Review both changes.
3. Keep the correct code.
4. Remove conflict markers.
5. Commit the resolved file.

```bash
git add file.txt
git commit
```

---

Q6: You accidentally deleted a branch that contained important commits. How would you recover it?

A: I would use `git reflog` to find the commit and recreate the branch.

```bash
git reflog
git checkout -b recovered-branch <commit-id>
```

---

Q7: You committed sensitive information like a password. What would you do?

A: I would:

1. Remove the secret from the code.
2. Change or revoke the password immediately.
3. Rewrite Git history if necessary.
4. Push the cleaned repository.

The first priority is always to rotate the leaked credentials.

---

Q8: Your local branch is behind the remote branch. How would you update it?

A: I would pull the latest changes.

```bash
git pull origin main
```

Or:

```bash
git fetch origin
git merge origin/main
```

This updates my local branch with remote changes.

---

Q9: You need only one commit from another branch. How would you do it?

A: I would use `git cherry-pick`.

```bash
git cherry-pick <commit-id>
```

This copies only the selected commit into the current branch.

---

Q10: You pushed a commit with the wrong commit message. How would you fix it?

A: If it is the latest commit:

```bash
git commit --amend -m "Correct message"
git push --force
```

I use force push carefully and preferably only on personal branches.

---

Q11: A production issue occurs and needs an immediate fix. What Git workflow would you follow?

A: I would:

1. Create a hotfix branch from the production branch.
2. Fix the issue.
3. Test it.
4. Merge it back to main and other active branches.

```bash
git checkout main
git checkout -b hotfix
```

---

Q12: You want to see who changed a particular line of code. What would you do?

A: I would use:

```bash
git blame file.txt
```

It shows the author, commit ID, and date for each line.

---

Q13: Your feature branch has many commits, and you want to keep history clean before merging. What would you do?

A: I would squash the commits.

```bash
git rebase -i HEAD~5
```

I can combine multiple commits into one meaningful commit.

---

Q14: You want to temporarily save changes in one file only. What would you do?

A: I would stash that specific file.

```bash
git stash push file.txt
```

This saves changes for only that file.

---

Q15: You accidentally ran `git reset --hard`. Can the changes be recovered?

A: Sometimes yes. I would check:

```bash
git reflog
```

If the commits still exist in the reflog, I can restore them.

---

Q16: You need to compare changes between two branches. How would you do it?

A: I would use:

```bash
git diff branch1 branch2
```

This shows the differences between the branches.

---

Q17: A merge created unnecessary commits and messy history. What would you use instead?

A: I would use rebase.

```bash
git checkout feature
git rebase main
```

Rebase creates a cleaner and linear commit history.

---

Q18: A file should no longer be tracked by Git but must remain on your computer. What would you do?

A: I would remove it from Git tracking.

```bash
git rm --cached file.txt
git commit -m "Stop tracking file"
```

The file stays locally but is removed from the repository.

---

Q19: You need to know which files changed in the last commit. What would you do?

A: I would run:

```bash
git show --name-only
```

This displays the files modified in the latest commit.

---

Q20: A teammate force-pushed and your local branch history no longer matches the remote branch. What would you do?

A: First, I would fetch the latest changes.

```bash
git fetch origin
```

Then I would review the differences and reset if necessary.

```bash
git reset --hard origin/main
```

I would ensure my local changes are backed up before using `reset --hard`.

---

Q21: You need to release version 1.0 of your application. How would you mark that version in Git?

A: I would create a tag.

```bash
git tag -a v1.0 -m "Version 1.0 Release"
git push origin v1.0
```

Tags help identify important release versions.

---

Q22: You accidentally committed directly to the main branch instead of creating a feature branch. What would you do?

A: I would create a new branch from my current state.

```bash
git branch feature-branch
git checkout feature-branch
```

Then I would remove the commit from main if it has not been pushed.

---

Q23: Before merging your branch, your manager asks you to update it with the latest main branch changes. What would you do?

A: I would:

```bash
git checkout feature
git fetch origin
git rebase origin/main
```

This keeps my branch updated and reduces merge conflicts.

---

Q24: You need to undo your last commit but keep the code changes. What would you do?

A: I would use:

```bash
git reset --soft HEAD~1
```

The commit is removed, but the code changes remain staged.

---

Q25: You need to find when and why a bug was introduced in the project. What would you do?

A: I would use:

```bash
git bisect start
git bisect bad
git bisect good <commit-id>
```

Git automatically helps identify the commit that introduced the bug, making debugging much faster.
